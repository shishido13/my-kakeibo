# My Kakeibo (家計簿アプリ) - システム仕様書

本ドキュメントは、現時点での「My Kakeibo」アプリケーションのシステム構成、技術スタック、アーキテクチャ、および実装済みの機能について整理・文書化したものです。今後の拡張やメンテナンスの際のリファレンスとして使用します。

---

## 1. システム全体構成

本システムは、モダンなWeb開発アーキテクチャである **SPA (Single Page Application) フロントエンド** と **RESTful API バックエンド** を分離して構築されています。

*   **フロントエンド**: ユーザーが直接操作する画面UIや状態管理を担当。
*   **バックエンド**: データの永続化（DB保存）、バリデーション、集計処理、外部サービス（AI等）連携などのビジネスロジックを担当。

---

## 2. 技術スタック

### 2.1. フロントエンド
*   **コアフレームワーク**: Vue 3 (Composition API, `<script setup lang="ts">`)
*   **開発用サーバー・ビルドツール**: Vite
*   **言語**: TypeScript
*   **状態管理**: Pinia
*   **ルーティング**: Vue Router
*   **HTTP クライアント**: Axios (バックエンドへのリクエスト送受信)
*   **スタイリング**: Tailwind CSS (ユーティリティファーストCSS)

### 2.2. バックエンド
*   **コアフレームワーク**: FastAPI
*   **言語**: Python 3.10+
*   **ORM (Object-Relational Mapping)**: SQLModel (Pydantic と SQLAlchemy のハイブリッド)
*   **通信定義・バリデーション**: Pydantic
*   **データベース**: SQLite (ローカルファイル `backend/db_data/kakeibo.db` への保存)
*   **環境変数管理**: `pydantic-settings`
*   **実行サーバー**: Uvicorn

## 3. ディレクトリ構成とアーキテクチャ

### 3.1. バックエンド (`/backend`)
ドメイン駆動や関心事の分離（Separation of Concerns）を意識したディレクトリ構造になっています。

```text
backend/
├── app/
│   ├── api/v1/         # ルーター層 (APIエンドポイントの定義)
│   ├── core/           # 設定ファイル (CORS, 環境変数など)
│   ├── crud/           # データベースへのCRUD操作を抽象化した層
│   ├── db/             # DBセッション管理と初期化スクリプト
│   ├── models/         # SQLModelによるテーブルスキーマ定義 (ORM)
│   ├── schemas/        # API入出力のバリデーション用Pydanticモデル
│   └── services/       # ビジネスロジック、AI解析や集計など複雑な処理を担う層
└── main.py             # FastAPIアプリケーションのエントリーポイント
```

#### 実装済み API エンドポイント
*   `/api/v1/transactions`: 取引履歴データのCRUD
*   `/api/v1/categories`: カテゴリのCRUD
*   `/api/v1/payers`: 支払者データのCRUD
*   `/api/v1/analytics`: ダッシュボード等で利用する統計・集計ロジック (`GET /summary/monthly`）
*   `/api/v1/receipts`: ファイルアップロードによるレシート解析 (`POST /analyze`, 現状AI模擬モック)

### 3.2. フロントエンド (`/frontend`)
コンポーネント指向アーキテクチャを採用し、UIの再利用性と状態管理を明確に分けています。

```text
frontend/
├── src/
│   ├── components/     # 再利用可能なUI部品 (TransactionList.vue, TransactionForm.vue など)
│   ├── router/         # Vue Router設定 (URLとViewの連携)
│   ├── services/       # Axios等のAPI通信設定 (バックエンドのURL定義など)
│   ├── stores/         # Piniaによるグローバル状態管理 (APIとのデータ同期・保持)
│   ├── views/          # ページごとのメインビュー (DashboardView.vue など)
│   ├── App.vue         # Vueのルートコンポーネント (全体のレイアウト基盤)
│   └── main.ts         # Vueアプリの起動スクリプト (プラグイン登録など)
├── index.html          # エントリーHTMLファイル
└── tsconfig.*.json     # TypeScript関連の設定ファイル群
```

## 4. データモデル (リレーショナルスキーマ)

SQLiteデータベース内には、以下の3つの主要テーブルが存在します。

1.  **Transaction (履歴データ)**
    *   中心となる取引データ。金額（`amount`）、日付（`date`）、店舗名（`shop`）、購入内容（`content`）、などの情報を保持。
    *   `category_id` を通じて Category テーブルとリレーションを持つ。
2.  **Category (カテゴリ)**
    *   「食費」「日用品」などのカテゴリマスタ。`is_active` フラグで論理削除を管理。
3.  **Payer (支払者)**
    *   「自分」「家族」などの支払者情報を管理するマスタ。こちらも `is_active` フラグ有り。

## 5. 今後の機能拡張性 (Service層の活用)

本システムは将来的な機能追加を想定し、初期から **Service層** (`backend/app/services/`) を導入しています。
単なるデータの右から左への移動（DB ⇔ API）は CRUD 層が担当し、以下のような複雑な機能はこの Service 層にて開発を進める設計思想です。

*   **生成AI連動**: `ai_service.py` を用いたレシートのOCR解析や、自動カテゴリ分類。
*   **高度な集計分析**: `transaction_service.py` を展開し、年間の推移グラフや異常支出の検知アルゴリズム実装。
*   **データインポート/エクスポート**: CSV や銀行APIを介した一括インポート機能の中間処理。
