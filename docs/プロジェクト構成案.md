
## **プロジェクトの全体像**

* **形態:** 全てローカル環境で実行するWebアプリケーション。
* **目的:** 自分専用の家計簿。将来的にはAIによる分析・入力を導入。
* **開発手法:** 仕様駆動開発（AIにプロンプトを投げてコード生成させる）。

## **技術スタック**

* **Frontend:** **Vue.js 3**（Vite, Composition API, Tailwind CSS, Chart.js）
* **Backend:** **Python / FastAPI**
* **Database:** **SQLite**
* **ORM:** **SQLModel**

## **プロジェクト構成**

* バックエンド（FastAPI）とフロントエンド（Vue.js）を分離した構成。
* 将来のAIロジックを切り出せるよう `services/` ディレクトリを配置。
```
my-kakeibo/
├── backend/                # FastAPI (Python)
│   ├── app/
│   │   ├── main.py         # エントリポイント
│   │   ├── api/            # APIルート定義 (v1/transactions.py 等)
│   │   ├── core/           # 設定、共通変数
│   │   ├── crud/           # DB操作ロジック (Create, Read, Update, Delete)
│   │   ├── models/         # SQLAlchemy のテーブル定義
│   │   ├── schemas/        # Pydantic のバリデーション定義
│   │   ├── db/             # SQLite 接続設定・初期化
│   │   └── services/       # ★将来のAIロジック、データ分析ロジック
│   ├── tests/              # テストコード
│   ├── kakeibo.db          # SQLite データベースファイル
│   └── requirements.txt    # 依存ライブラリ
├── frontend/               # Vue.js 3 (Vite)
│   ├── src/
│   │   ├── assets/         # 画像、CSS (Tailwind)
│   │   ├── components/     # 再利用可能なUIパーツ (Chart, Form)
│   │   ├── views/          # ページ全体 (Dashboard, History)
│   │   ├── services/       # API通信 (Axios等)
│   │   ├── stores/         # 状態管理 (Pinia)
│   │   └── App.vue
│   └── package.json
└── docker-compose.yml      # ローカル一括起動用
```

##  データベース設計 (SQLModel)

### `t_transaction` (明細)

| フィールド名 | 型 | 説明 |
| --- | --- | --- |
| `id` | int (PK) | 自動採番 |
| `date` | date | 支出が発生した日付 |
| `amount` | int | 金額 |
| `category_id` | int (FK) | `Categories.id` への外部キー |
| `shop` | str | 店舗名 |
| `content` | str | 商品・サービス内容 |
| `payer`| str| 支払者 |
| `description` | str (nullable) | 備考、メモ |
| `source_type` | str | 登録元 ("manual", "csv", "ocr" 等) |
| `created_at` | datetime | システム登録日時 (自動生成) |

### `m_category` (カテゴリ)

| フィールド名 | 型 | 説明 |
| --- | --- | --- |
| `id` | int (PK) | 自動採番 |
| `name` | str | カテゴリ名（食費、日用品など） |
| `is_active` | bool | 論理削除用フラグ（初期値: True） |

### `m_payer` (支払者)

| フィールド名 | 型 | 説明 |    
| --- | --- | --- |
| `id` | int (PK) | 自動採番 |
| `name` | str | 支払者名 |
| `is_active` | bool | 論理削除用フラグ（初期値: True） |

##  画面構成 (UI/UX)

### **メイン画面（履歴一覧）**

* **上部:** 「新規登録」ボタン（モーダル表示）
* **検索・フィルタエリア:** * 期間検索（カレンダーによる範囲指定）
* フリーワード検索（`shop`, `content`, `description` を対象）
* カテゴリ絞り込み（プルダウン）


* **一覧テーブル:** * 最新順（日付降順）に表示。
* 各行に編集・削除機能を備える。



### **入力モーダル**

* 入力項目：日付（デフォルト当日）、金額、カテゴリ選択、店舗、商品、備考。
* 保存後、続けて入力ができる「連続入力モード」のオプションがあると望ましい。


## バックエンド構成方針

* **ディレクトリ構成:** `api/` (ルーティング), `crud/` (DB操作), `models/` (型定義), `services/` (ビジネスロジック) に分離。
* **将来のAI拡張:** AI APIを叩く処理や複雑な解析は `services/` に集約し、APIエンドポイントから独立させる。
* **CSV/OCR拡張:** 将来的に一括登録を実装するため、リスト形式のデータを受け入れる `bulk_create` 用のロジックを `crud` 層に想定しておく。
